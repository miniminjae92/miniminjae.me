# SSR, CSR, SPA를 나눠 부르는 진짜 이유

처음에는 SSR, CSR, SPA가 그저 “용어 지옥”처럼 느껴졌다. 하지만 결국 이것들도 하나의 질문에서 출발한다.

> “누가, 언제, 어디서 HTML을 만들 것인가?”

## SSR(Server-Side Rendering)

- 서버에서 HTML을 만들어 통째로 내려보낸다.
- 장점: 초기 로딩 속도, SEO(Search Engine Optimization(검색 엔진 최적화))에 유리
- 단점: 페이지 전환마다 새 요청, 상태 유지가 번거로울 수 있음

## CSR(Client-Side Rendering)

- 서버는 JSON 위주의 데이터만 제공한다.
- 브라우저가 JS로 화면을 렌더링한다.
- 장점: 페이지 전환이 부드럽고, 앱처럼 동작
- 단점: 최초 로딩이 무거워질 수 있다.

## SPA(Single Page Application(단일 페이지 애플리케이션))의 목적

SPA는 “문서”보다는 “앱 같은 경험”을 만들기 위한 선택이었다.

- URL은 바뀌지만, 브라우저는 새로고침 없이 JS로 화면을 갈아끼운다.
- 사용자는 “화면 이동”이 아니라 “상태 변화”로 느끼게 된다.

## Next.js 같은 프레임워크가 해주는 일

Next.js는 SSR과 CSR, 그리고 그 사이 여러 전략을 조합해서 쓰도록 도와준다.

- 일부 페이지는 SSR, 일부는 CSR
- 정적 페이지, 서버 컴포넌트, 클라이언트 컴포넌트를 섞어서 사용

그래서 이제는 이렇게 정리할 수 있다.

> SSR과 CSR은 싸우는 개념이 아니라,  
> **상황에 맞게 섞어 쓰는 재료**에 가깝다.
