# 클라이언트·서버·JSON, 웹앱의 최소 단위

웹앱을 이해하려 할 때, 처음에는 “화면이 어떻게 보이는지”에만 집중했다. 하지만 점점 느끼는 건, 화면보다 **데이터 흐름**을 이해하는 게 더 중요하다는 점이다.

## 클라이언트와 서버의 역할

- **클라이언트(Client)**: 브라우저처럼 요청을 보내고 결과를 보여주는 쪽  
- **서버(Server)**: 요청을 받고, 처리해서 응답을 돌려주는 쪽

핵심은 단순하다.

> 클라이언트는 “이거 좀 해줘요”라고 말하고,  
> 서버는 “이렇게 처리했습니다”라고 답한다.

## HTML 전체 vs JSON만 주고받기

예전에는 서버가 매번 **완성된 HTML**을 내려줬다.  
지금의 많은 웹앱에서는 서버가 **JSON(JavaScript Object Notation)**으로 데이터만 내려주고, 브라우저가 그걸 화면으로 렌더링한다.

- 서버: `{"title": "내 첫 글", "content": "..."}` 같은 JSON 반환  
- 클라이언트: 이 JSON을 받아 React/Next.js로 화면을 그린다.

## “언제 HTML로 바꾸느냐”의 문제

- 서버가 HTML까지 만들어 보내면: 전통적인 SSR 느낌  
- 서버는 JSON만, HTML은 브라우저에서 만들면: CSR 중심

즉, 우리는 이렇게 묻는 셈이다.

> “데이터를 **어디서** 화면(HTML)으로 바꿀 것인가?”

이 질문이 정리되면, 프론트와 백엔드를 같은 언어로 이야기할 수 있다.
